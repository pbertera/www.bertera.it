Spawning a network namespace with sudo

Posted on Mon 31 January 2022 in Systems

After 1 year and half a new post! I'm improving :)

This post is to keep track of some subtle issues that took me some hours to face and solve.

The intention

In a setup I needed to script the creation of two unnamed Kernel network namespaces and connect them with a veth network interfaces pair.

This is pretty a simple thing, as root on a system:

  • Spawn two background processes holding the netns and gather the PID:
# unshare --net -- sleep inf &
# netns1_pid=$!
# unshare --net -- sleep inf &
# netns2_pid=$!
  • Create the veth pairs:
# ip link add veth1 type veth peer name veth2
  • Assign the veth pairs to the netns:
# ip link set veth1 netns $netns1_pid up
# ip link set veth2 netns $netns2_pid up
  • Assign them an IP address:
# nsenter -n -t $netns1_pid ip addr add 192.168.1.10/24 dev veth1
# nsenter -n -t $netns2_pid ip addr add 192.168.1.20/24 dev veth2
  • Test the connectivity between the two netns
# nsenter -n -t $netns1_pid ip addr
# nsenter -n -t $netns2_pid ip addr
# nsenter -n -t $netns2_pid ping 192.168.1.10 -c 3

This works flawlessly on a root terminal. Putting everything in a script raised some issues.

The subshell blocks spawning a new namespace

First I put the netns creation in a function:

# cat ns.sh
#!/bin/bash

set -x

function spawnNS {
  unshare --net -- sleep inf &
  echo $!
}

netns1_pid=$(spawnNS)
echo "New netns spawn: $netns1_pid"

# ./ns.sh 
++ spawnNS
++ echo 83586
++ unshare --net sleep inf

Now: the ns.sh script never terminates until you kill the process (83586 here). Ok, this has nothing to do with namespaces: is a known shell behaviour with command substiution. Command substitution creates a pipe in the main shell, the subshell where spawnNS is executed owns the writing end of that pipe: the main shell remains blocked because it never gets an EOF from the pipe. Redirecting the output of unshare solves the issue:

#!/bin/bash

set -x

function spawnNS {
  unshare --net -- sleep inf >/dev/null &
  echo $!
}

netns1_pid=$(spawnNS)
echo "New netns spawn: $netns1_pid"

Another solution is to avoid command substitution when invoking the spawnNS function:

#!/bin/bash

set -x

ns_pid=""
function spawnNS {
  ns_pid=""
  unshare --net -- sleep inf &
  ns_pid=$!
}

spawnNS
netns1_pid=$ns_pid
echo "New netns spawn: $netns1_pid"

Dealing with sudo

I wanted to run the script from an unprivileged user and use sudo to invoke unshare:

$ ./ns.sh
++ spawnNS
++ echo 110799
++ sudo unshare --net -- sleep inf
+ netns1_pid=110799
+ echo 'New netns spawn: 110799'
New netns spawn: 110799

Now, let's check the networking within the netns:

$ sudo nsenter -t 110799 -n ip -o a

nsenter returns all the host network interfaces: something is wrong here!

$ pstree -apl 110799
sudo,110799 unshare --net -- sleep inf
  └─sleep,110805 inf

Oh yes, sudo forks, so $! returns the sudo PID, so let's grab the children PID:

$ cat ns.sh
#!/bin/bash                              

set -x                                   

function spawnNS {                       
  sudo unshare --net -- sleep inf > /dev/null &
  ns_pid=$(ps --ppid $! -o pid=)
  echo $ns_pid
}                                        

netns1_pid=$(spawnNS)                    
echo "New netns spawn: $netns1_pid"

$./ns.sh
++ spawnNS
++ sudo unshare --net -- sleep inf
+++ ps --ppid 112780 -o pid=
++ ns_pid=
++ echo
+ netns1_pid=
+ echo 'New netns spawn: '
New netns spawn:

Even if the namespace has been created something is still wrong: the PID of sleep is not reported.

$ sudo lsns -t net
        NS TYPE NPROCS    PID USER      NETNSID NSFS COMMAND
4026532836 net       1 113284 root   unassigned      sleep inf

After some digging with strace I found out that the reason was a race condition between the unshare and pid processes: pid --ppid $! spawn too quickly and the sudo process did not fork yet. Adding a sufficient delay solved the problem:

$ cat ns.sh
#!/bin/bash                              

set -x                                   

function spawnNS {                       
  sudo unshare --net -- sleep inf > /dev/null &
  ppid=$!
  sleep 0.1
  ns_pid=$(ps --ppid $ppid -o pid=)
  echo $ns_pid
}                                        

netns1_pid=$(spawnNS)                    
echo "New netns spawn: $netns1_pid"

$ ./ns.sh
++ spawnNS
++ ppid=114064
++ sleep 0.1
++ sudo unshare --net -- sleep inf
+++ ps --ppid 114064 -o pid=
++ ns_pid=' 114067'
++ echo 114067
+ netns1_pid=114067
+ echo 'New netns spawn: 114067'
New netns spawn: 114067

So here is the full code snippet to setup two network namespaces and connecting them with a veth pair

#!/bin/bash

function bailout {
    echo "ERROR: $@"
    exit
}

function newNetNs {
    [ "$1" == "" ] && bailout "$0 requires a netns name"
    name=$1
    sudo unshare --net --uts -- sleep inf & 
    i=0
    ns_pid=""
    while [ "$ns_pid" == "" ]; do
        ns_pid=$(ps --ppid $! -o pid=)
        sleep 0.01 
        i=$(($i + 1 ))
        [ $i -gt 10 ] && bailout "Cannot get the PID of namespace $name"
    done
    sudo nsenter -t $ns_pid -u hostname $name
}

newNetNs netns1
netns1_pid=$ns_pid
echo "Created netns1 $netns1_pid"
newNetNs netns2
netns2_pid=$ns_pid
echo "Created netns2 $netns2_pid"

sudo ip link add veth1 type veth peer name veth2
sudo ip link set veth1 netns $netns1_pid up
sudo ip link set veth2 netns $netns2_pid up
sudo nsenter -n -t $netns1_pid ip addr add 192.168.1.10/24 dev veth1
sudo nsenter -n -t $netns2_pid ip addr add 192.168.1.20/24 dev veth2

sudo nsenter -n -t $netns2_pid ping 192.168.1.10 -c 3

# Destroy everything
sudo kill -9 $netns1_pid $netns2_pid
sudo ip link delete veth1

Putting together those lines of bash took me a full morning, as usual I understimated the complexity (or overstimated myself).
